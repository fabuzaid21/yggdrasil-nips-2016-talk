Distributed Decision Trees: The Classic Approach (PLANET)

  • Partition training set by row
  • Each worker computes sufficient statistics on subset of training data
  • Since data partitioned by row, workers must compute statistics for all features
  • Communication cost: $O(2^DkpB)$
  • To compute the split for a given node:
    • Each worker computes sufficient statistics on the subset of training data available to it
    • Since data is partitioned by row, workers must compute statistics for all features
    • The master aggregates the statistics, then picks the best feature (and threshold) to split on

The Problem with Partitioning by Row

  • To limit communication, workers only consider $B$ thresholds, instead of $n-1$
  • Optimal split may not always be found!
  • Key takeaway:
    Partitioning by row
    1. introduces a tradeoff between accuracy and runtime efficiency
    2. asks the user to tune an additional hyperparameter $B$
Excerpt from page 5 of the PLANET paper
  • The thresholds are found by computing a histogram per feature at the beginning of the algorithm.
  • This means that the optimal split may not always be found, especially for continuous features!

Big Idea: Partition by column

The New Approach: Yggdrasil

  • Each worker computes sufficient statistics on all of its local features and sends the best feature to the master
  • No $B$ parameter; all thresholds considered
  • Communication cost: $O(2^Dk + Dnk)$
  • For each node:
    • The master picks the globally best feature to split on from the $k$ candidates

Yggdrasil in Action

  1. Partition features across workers
  2. Workers sort each feature by value
  3. Compute best split for each feature
  4. Pick best split for each worker, send split + bit vector to master
  5. Master selects best global split among the candidates
  6. Master sends bit vector for best global split to each worker
  7. Worker sorts each feature by bit vector (using label indices), then value

Why sort?

  • Must keep track of the split history for each feature
  • New column per split $\rightarrow$ exponential memory overhead
  • With sorting, memory overhead is constant

How expensive is sorting?

Does Yggdrasil do an $O(n \log n)$ sort per iteration?

No!

  • The first sort operation is $O(n \log n)$
  • Sorting based on bit vector is $O(n)$, since the data is already sorted

Optimizations

  • Sparse bitvectors
  • Label encoding
  • Columnar compression: Run-Length Encoding
